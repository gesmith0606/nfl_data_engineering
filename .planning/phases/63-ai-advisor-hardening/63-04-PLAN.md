<objective>
Make ADVR-02 — "Asking 'who are the top 10 RBs' returns a ranked list with real projected points from the Gold layer" — work reliably. The tool already exists (`getPositionRankings`). The reliability gap is that the LLM must guess which `season` + `week` to request; when it guesses wrong, the tool returns 404 or an empty list and the advisor hallucinates.

Purpose: Ground the position-rankings flow in real Gold data by (a) auto-resolving the latest available week server-side, (b) surfacing the `data_as_of` timestamp in the response, and (c) adding a contract test that catches any regression where `projected_points` drops to zero or the sort order inverts.

Output: Frontend tool improvement + a contract test ensuring the Gold layer actually powers the response.
</objective>

<interfaces>
<!-- getPositionRankings tool contract (current) -->

tool.getPositionRankings.inputSchema:
  position: 'QB'|'RB'|'WR'|'TE'|'K'
  limit:    number (1..50, default 10)
  season:   number (default 2026)
  week:     number (REQUIRED, 1..18)   ← the reliability hole
  scoring:  'ppr'|'half_ppr'|'standard' (default half_ppr)

Current implementation calls GET /api/projections?position=...&limit=...&season=...&week=...
and sorts response.projections by projected_points descending.

Gold layer file pattern:
  data/gold/projections/season=YYYY/week=WW/projections_YYYYMMDD_HHMMSS.parquet
  Preseason: data/gold/projections/season=YYYY/week=0/ (week=0 is the preseason slate)

What the LLM is doing wrong today:
  When the user says "who are the top 10 RBs" without specifying a week, Gemini
  usually fills in week=1. If season=2026/week=1 has no data yet (in real 2026
  offseason this happens), the endpoint returns {} or an empty list and the
  model hallucinates rankings.
