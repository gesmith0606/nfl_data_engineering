<objective>
Wire the rule-extracted structured events into the projection engine per D-03: add `apply_event_adjustments()` as a companion to `apply_injury_adjustments()`, mapping each event flag to a deterministic, tightly-bounded multiplier. Then backtest 2022–2024 half_ppr and gate SHIP on non-regression vs the 5.05 MAE production baseline.

Purpose: D-03 forbids piping a continuous 0.70–1.15 sentiment_multiplier into projections (Phase 54 taught us wide ranges degrade production even when WFCV looks good). Instead, each structured event gets a specific bounded multiplier tied to a specific observable behavior — reproducible, backtestable, easy to audit.

Output: A new adjustment layer in the projection engine, a backtest script, a test suite, and a human checkpoint for the SHIP/SKIP decision.
</objective>

New companion function signature this plan creates:

```python
def apply_event_adjustments(
    projections_df: pd.DataFrame,
    events_df: pd.DataFrame,
) -> pd.DataFrame:
    """
    Returns: Projections DataFrame with added columns:
        - event_flags: List[str] (human-readable tags for downstream display)
        - event_multiplier: float (product of all event multipliers for that player)
    projected_points and proj_* columns are scaled by event_multiplier.
    """
```

events_df source: Gold sentiment Parquet written by
`src/sentiment/aggregation/weekly.py::WeeklyAggregator`. After Plan 02, every
event flag added to PlayerSignal propagates through _build_silver_record →
Silver JSON → WeeklyAggregator → Gold Parquet columns named identically
(`is_questionable`, `is_returning`, `is_traded`, `is_released`, `is_signed`,
`is_activated`, `is_usage_boost`, `is_usage_drop`, `is_weather_risk`,
`is_ruled_out`, `is_inactive`, `is_suspended`).

Join key: `player_id` (canonical gsis_id), matching how
`apply_injury_adjustments` joins. Gold sentiment already resolves player_id via
PlayerNameResolver.

Deterministic multiplier table (per D-03 — tightly bounded, NOT continuous):

| Event flag         | Multiplier | Rationale |
|--------------------|-----------|-----------|
| is_ruled_out       | 0.00      | player doesn't play (redundant w/ injury but safety net) |
| is_inactive        | 0.00      | player doesn't play |
| is_questionable    | 0.85      | matches existing INJURY_MULTIPLIERS[Questionable] |
| is_suspended       | 0.00      | player doesn't play |
| is_returning       | 0.90      | first game back — conservative |
| is_activated       | 0.90      | same as returning |
| is_traded          | 0.85      | new team uncertainty |
| is_released        | 0.00      | no team, no projection |
| is_signed          | 1.00      | neutral; the player has a team, no usage signal yet |
| is_usage_boost     | 1.08      | bounded upside |
| is_usage_drop      | 0.85      | bounded downside |
| is_weather_risk    | 0.92      | game-level dampener |

When multiple flags are true, multiply them together and clamp to [0.0, 1.10]
to prevent compounding beyond the bounds of trust.
