### Phase 76: AWS Refresh + S3 Sync
**Goal**: Cloud storage layer is reachable and current; production reads can resume on S3 when needed and PFF prep (v8.0) has a clean foundation.
**Depends on**: Nothing (independent of all other v7.2 work).
**Requirements**: AWS-01, AWS-02, AWS-03
**Success Criteria** (what must be TRUE):
  1. Operator can read latest Bronze/Silver/Gold parquet from S3 in a fresh shell using `download_latest_parquet()` against all three buckets (`nfl-raw`, `nfl-refined`, `nfl-trusted`).
  2. Local `data/bronze/`, `data/silver/`, `data/gold/` are mirrored to S3 in a one-shot sync; re-running the sync writes zero new objects (idempotent).
  3. CI smoke job runs at least one production read path (Silver transformation script) against S3 and passes when AWS_* secrets are configured; skips gracefully when not.

### Phase 77: Sentiment Source Expansion
**Goal**: Daily sentiment cron pulls from 7 sources instead of 5 so team-event coverage floor lifts above the current 9-12 teams toward the 15-20 range.
**Depends on**: Nothing (independent of all other v7.2 work).
**Requirements**: INGEST-01, INGEST-02, INGEST-03
**Success Criteria** (what must be TRUE):
  1. ESPN team RSS ingester writes one Bronze JSON per run to `data/bronze/sentiment/espn_team/season=YYYY/` and is wired into `daily-sentiment.yml` under the D-06 fail-open contract.

### Phase 78: Heuristic Consolidation
**Goal**: All three legacy heuristic call sites delegate to one canonical function; the v4.1-p3 QB-bias regression cannot recur.
**Depends on**: Nothing (independent; pure refactor + tests).
**Requirements**: HEUR-01, HEUR-02, HEUR-03
**Success Criteria** (what must be TRUE):
  1. A single canonical `compute_heuristic_projection()` (or equivalent) exists with golden-output fixtures pinned for at least 5 representative player-weeks per position (QB/RB/WR/TE).
  2. `generate_weekly_projections` (projection_engine.py), `generate_heuristic_predictions` (player_model_training.py), and `compute_production_heuristic` (unified_evaluation.py) all call the canonical function; behavioral equivalence is proved by golden tests passing for every legacy caller.
  3. A QB-bias regression test asserts the QB heuristic does not NaN-propagate when `usage_multiplier` is missing or stale upstream (the v4.1-p3 root cause).

### Phase 79: Audit Provenance + Live Version Probe
**Goal**: Every audit-JSON artifact and every running deploy can be traced to the exact code commit that produced it; the live deploy gate can no longer pass against a stale image.
**Depends on**: Nothing (Phase 80 and Phase 84 consume its outputs but do not block its work).
**Requirements**: DQ-01, DQ-02
**Success Criteria** (what must be TRUE):
  1. Audit-script JSON outputs (`event_coverage.json`, `advisor_tools_72.json`, and any future audit JSONs) include a `script_sha` field capturing the audit script's git SHA at execution time.
  2. Railway exposes `/api/version` returning `{git_sha, deployed_at, llm_enrichment_ready}` reflecting the running image (not a hardcoded value).
  3. A live curl against `/api/version` after a fresh push returns the GITHUB_SHA of the just-pushed commit (proves the asymmetry-detection capability the 3-week silent freeze required).

### Phase 80: Sanity Warning Triage
**Goal**: All 27 pre-existing sanity-warning items have a documented verdict; no warning is left as silent debt.
**Depends on**: Nothing (parallelizable with all other phases).
**Requirements**: DQ-03
**Success Criteria** (what must be TRUE):
  1. A triage document under `.planning/milestones/v7.2-phases/[phase]/DQ-03-TRIAGE.md` lists every one of the 27 sanity warnings with one of four verdicts: KEEP / UPGRADE / DELETE / RAISE-THRESHOLD.
  2. Every UPGRADE verdict has a follow-up commit promoting the check from WARNING to CRITICAL.
  3. Every DELETE verdict has a follow-up commit removing the stale check; every RAISE-THRESHOLD verdict has a commit relaxing the threshold with a documented "why" comment.
  4. Post-triage `python scripts/sanity_check_projections.py --check-live` produces a warning count consistent with the triage outcome (KEEP-only count).

### Phase 82: Sleeper OAuth + Multi-User
**Goal**: Users can sign in via Sleeper OAuth, sessions persist across 24h gaps, and the existing cookie-based path keeps working for backwards-compat.
**Depends on**: Nothing (independent; consumed by Phase 83 TOOL-03).
**Requirements**: AUTH-01, AUTH-02, AUTH-03
**Success Criteria** (what must be TRUE):
  1. User can sign in via Sleeper OAuth; the resulting refresh token + session row are persisted to a backend session table (SQLite or Postgres). The browser cookie carries an opaque session ID, never the raw Sleeper user_id.
  2. A user with a 24h-stale session can hit any auth-gated endpoint and the refresh-token exchange transparently renews the session with no re-auth prompt; refresh failure returns a clean re-auth redirect, not a 500.
  3. The existing cookie-based `sleeper_user_id` flow from Phase 74 still resolves leagues/rosters when no OAuth session exists (backwards-compat path remains green).
**Plans**: TBD
**UI hint**: yes

### Phase 83: Advisor Tool Expansion
**Goal**: The chat advisor exposes the v7.1-shipped surfaces (external projections, matchup advantages, user rosters) as native tools the LLM can call without prompt-engineering hacks.
**Depends on**: Phase 82 (TOOL-03 needs auth-aware session lookup).
**Requirements**: TOOL-01, TOOL-02, TOOL-03
**Success Criteria** (what must be TRUE):
  1. `getProjectionComparison({player_name})` returns the 4-source comparison table for a player; live test against the deployed advisor confirms the LLM calls it for "what does ESPN say about Player X" prompts.
  2. `getMatchupAdvantages({team, week})` returns positional-rank advantages for a team-week wrapping `/api/teams/{team}/defense-metrics`; live test confirms the LLM calls it for "how good is X's defense" prompts.
  3. `getUserLeagueRoster` is auth-aware — uses the OAuth session from Phase 82 when present and falls back to the Phase 74 cookie-based `sleeper_user_id` when not; both paths return a roster, the tool list does not bifurcate.

### Phase 84: Deploy Hardening
**Goal**: The deploy gate detects asymmetric drift, rejects audit evidence from unknown script revisions, survives workflow-touching rollbacks, and is structurally locked against future regressions.
**Depends on**: Phase 79 (DEPLOY-02 needs `/api/version`; DEPLOY-04 needs script_sha).
**Requirements**: DEPLOY-01, DEPLOY-02, DEPLOY-03, DEPLOY-04
**Success Criteria** (what must be TRUE):
  1. Auto-rollback either skips the push (with `::error` summary listing manual-rollback steps) when the failing commit touches `.github/workflows/**`, or uses a provisioned `ROLLBACK_PAT` with `workflow` scope; structural test pins whichever path is chosen.
  2. Vercel chunk-fingerprint probe in `deploy-frontend` is fail-on-mismatch when the fingerprint is unchanged AND `/api/version` shows Railway redeployed (asymmetric drift = real Vercel webhook freeze, no longer warn-only).
  3. A structural test asserts the rebase+retry loop in `daily-sentiment.yml` (commit `aa4ac70`) survives future refactors of the daily cron's push step.
  4. The live gate's audit-JSON consumer asserts `script_sha` exists and matches a known-good audit-script commit (or main HEAD); audit evidence produced by an unknown script revision is rejected.
