# v8.0 Live Draft Co-Pilot — Roadmap

**Goal:** A Claude Code-native live draft assistant across **Sleeper, Yahoo, and ESPN**. During a live draft on any supported platform, Claude polls picks in real time, tracks the full board + every team's roster, auto-detects the user's pick slot, and proactively surfaces VORP/projection-backed recommendations and key-moment alerts — entirely inside the VS Code session.

**Sleeper-first, then expand.** Sleeper ships the full vertical (85→86→87) and proves the loop; Yahoo and ESPN then plug into the same engine as additional adapters (88, 89). All three are poll-and-diff under the hood — the engine is built adapter-based so new platforms need no engine rewrite.

**Explicitly out of scope:** This is **separate from the website chatbot**. No frontend, no website integration, no client-facing delivery. The site chatbot keeps answering site users' questions; this co-pilot is a personal Claude Code tool for the operator on draft night.

**Phase range:** 85–89 (continues from v7.2's last phase 84).
**Total phases:** 5. Linear core (85→86→87 = Sleeper v1), then 88 (Yahoo) and 89 (ESPN) extend it.
**Total requirements:** 18 across 5 categories.

## Locked Decisions (from scoping conversation 2026-06-14)

- **D-01 — Delivery surface:** Claude Code / VS Code session only. The deliverable is a `/draft-live` skill + supporting Python, not a web feature.
- **D-02 — Platforms: Sleeper → Yahoo → ESPN, all in v8.0.** Sleeper ships first (no-auth public API, already have `src/sleeper_http.py`). Yahoo follows (official Fantasy API, OAuth2 + token refresh, poll `draft_results`). ESPN last and **spike-gated** — it has NO official live API, so a time-boxed feasibility spike (REST `mDraftDetail` latency vs Playwright/DOM watcher vs cookie auth) decides GO/NO-GO before any adapter is built.
- **D-08 — Adapter architecture.** Phase 86 defines a `DraftAdapter` protocol (resolve draft → poll picks → normalize to `PickEvent`/`DraftState` → map players → projections). `SleeperAdapter` is the reference implementation; `YahooAdapter` (88) and `EspnAdapter` (89) conform to the same interface so the engine + `/draft-live` skill run unmodified per platform. Each platform supplies its own auth + player-id mapping; everything downstream of the normalized pick event is shared.
- **D-09 — Manual-entry fallback.** When no adapter is available (unsupported platform, or an API/cookie breaks mid-draft), the skill supports operator-typed picks so a draft is never un-assistable. This is the guaranteed ESPN floor if the spike returns NO-GO.
- **D-03 — Live mechanism: poll-and-diff.** No public websocket. Poll `GET /v1/draft/{draft_id}/picks` ~every 2s, diff on `pick_no`, emit new pick events. Safe vs Sleeper's ~1000 req/min soft cap.
- **D-04 — Entry: username OR draft_id.** User can hand a draft_id, or just their Sleeper username/league and the tool resolves the active/most-recent draft.
- **D-05 — Proactivity: proactive on user's turn + key moments.** Auto-surface a recommendation when the user is on the clock; flag value drops / positional runs / reaches as opponents pick; quiet otherwise; user can ask anytime.
- **D-06 — Advisor = Claude (the session).** In VS Code, Claude is the advisor directly. The Gemini/Groq website chat stack is untouched and unrelated.
- **D-07 — Reuse-first.** Build on `src/draft_optimizer.py` (DraftBoard, DraftAdvisor, VORP, value tiers, auction, mock sim), `src/sleeper_http.py`, and existing projection/ADP access. Net-new is only the Sleeper draft endpoints, the live state tracker, and the skill wrapper.

## Phase Details

### Phase 85: Sleeper Draft Data Layer
**Goal:** All Sleeper draft data needed to drive a live draft is reachable through the existing fail-open HTTP wrapper, normalized into typed state, and the Sleeper player universe is mapped to our projections.
**Depends on:** Nothing.
**Requirements:** SLPR-01, SLPR-02, SLPR-03, SLPR-04
**Success Criteria** (what must be TRUE):
  1. `src/sleeper_http.py` exposes `get_drafts_for_league(league_id)`, `get_draft(draft_id)`, `get_draft_picks(draft_id)`, and `get_traded_picks(draft_id)`, each preserving the fail-open ({}/[]) contract on network/HTTP/parse error.
  2. Given a Sleeper username (+ season), the layer resolves the user's active or most-recent draft and returns its `draft_id`, scoring/roster settings, and draft order without any manual ID lookup.
  3. A Sleeper `player_id` → our projection player (name/position/team) mapping is built from the `/v1/players/nfl` registry, cached locally, and maps ≥ 95% of skill-position picks; unmatched picks (DST/K/obscure rookies) are surfaced, never silently dropped.
  4. Normalized `DraftState` and `PickEvent` models exist (pick_no, round, draft_slot, roster_id, player, team, position, is_keeper) with unit tests passing against a recorded real-draft fixture JSON.

### Phase 86: Live Draft Engine
**Goal:** Replaying or live-polling a Sleeper draft produces correct, continuously-updated board + roster state, knows whose turn it is and which slot is the user's, and emits projection-backed recommendations and key-moment alerts.
**Depends on:** Phase 85.
**Requirements:** ENG-01, ENG-02, ENG-03, ENG-04, ENG-05
**Success Criteria** (what must be TRUE):
  1. A poll/diff loop, given a `draft_id`, detects draft status (pre-draft / active / complete) and emits only new `PickEvent`s since the last-seen `pick_no` on each tick (idempotent; no duplicate emissions).
  2. Live state syncs a `DraftBoard` (available vs drafted), reconstructs every team's roster from the pick log, computes snake pick order + the on-the-clock slot, and identifies the user's draft slot and their next pick number.
  3. On the user's turn the engine returns DraftAdvisor top-N recommendations with reasoning (VORP, value tier vs ADP, roster need, bye-week conflict) and best-available-by-position, sourced from current projections.
  4. Key-moment detection fires correctly on constructed scenarios: value drop (a high-VORP player still available past expected), positional run (≥3 of one position in a short window), reach/steal vs ADP, and a post-pick grade for each completed pick.
  5. A `DraftAdapter` protocol is defined (`resolve`, `poll_state`/`poll_new_picks`, normalize → `PickEvent`/`DraftState`, `map_players`) and `SleeperAdapter` implements it wrapping Phase 85's `sleeper_draft.py`; the engine consumes ONLY the adapter interface, never Sleeper specifics directly (proves Yahoo/ESPN can slot in without engine changes).

### Phase 87: Claude Code Draft Skill
**Goal:** The operator runs a full live draft with Claude as a proactive co-pilot from one VS Code command, with no manual plumbing.
**Depends on:** Phase 86.
**Requirements:** SKILL-01, SKILL-02, SKILL-03, SKILL-04
**Success Criteria** (what must be TRUE):
  1. A `/draft-live` skill (SKILL.md + supporting entrypoint) starts from a `--platform` + username/draft_id, dispatches through the `DraftAdapter` interface (Sleeper live at this phase), auto-applies the league's scoring + roster format, and begins live polling — usable without the operator editing code.
  2. During the draft Claude presents evolving state + advice and answers ad-hoc questions ("who should I pick", "best WR left", "compare X vs Y", "what do I still need"), and per D-05 is proactive on the user's turn and on key moments while staying quiet otherwise.
  3. Operational polish: graceful start/stop, resume mid-draft (re-sync from full pick log), tolerant of paused/slow drafts, at least read-only awareness of auction drafts (budgets/nominations), with usage documented in CLAUDE.md and cross-linked from the existing `draft-prep` skill.
  4. Manual-entry fallback (D-09): the operator can drive the same engine by typing picks ("drafted: Bijan Robinson") when no adapter is live, so any draft on any platform is assistable even before/without Yahoo/ESPN adapters.

### Phase 88: Yahoo Draft Adapter
**Goal:** A `YahooAdapter` conforming to the Phase 86 `DraftAdapter` protocol streams live picks from a Yahoo league draft via the official Fantasy Sports API, so the engine + `/draft-live` skill work on Yahoo with zero engine changes.
**Depends on:** Phase 86 (adapter protocol), Phase 87 (skill dispatch + manual fallback).
**Requirements:** YH-01, YH-02, YH-03
**Success Criteria** (what must be TRUE):
  1. Yahoo OAuth2 completes once and the access token auto-refreshes (≈60-min expiry) across a full draft-length session with no re-auth prompt; client id/secret + tokens come only from env/`.env` (never committed). A clean re-auth path exists on refresh failure.
  2. `YahooAdapter` implements the same `DraftAdapter` interface as Sleeper (resolve league/draft → poll `draft_results` → normalize → map players); the unmodified engine + skill run end-to-end against a Yahoo draft. Polling is conservative with backoff (Yahoo's undocumented throttling / HTTP 999 risk).
  3. On a recorded `draft_results` fixture, picks normalize to `PickEvent`/`DraftState` and map ≥95% of skill-position players to projections; offline tests pass with OAuth + HTTP mocked (no live network in CI). Operator smoke against a real Yahoo draft documented.

### Phase 89: ESPN Draft Adapter (spike-gated)
**Goal:** Decide and, if viable, deliver live ESPN draft capture behind the same adapter interface — accepting that ESPN has no official live API.
**Depends on:** Phase 86, Phase 87. (Independent of Phase 88.)
**Requirements:** ESPN-01, ESPN-02, ESPN-03
**Success Criteria** (what must be TRUE):
  1. **[SPIKE — ESPN-01]** A time-boxed feasibility spike produces a documented GO/NO-GO verdict and a chosen live-capture mechanism for in-progress ESPN picks, with evidence — evaluating (a) `mDraftDetail` REST polling latency during a live draft, (b) a Playwright/DOM watcher of the draft room, and (c) `espn_s2`+`SWID` cookie harvest/refresh. The spike GATES ESPN-02/03.
  2. **[GO path — ESPN-02/03]** `EspnAdapter` conforms to `DraftAdapter` via the spike-chosen mechanism, behind a feature flag; the engine + skill run against a live ESPN draft; picks normalize and map ≥90% of skill-position players on a fixture; fragility + ToS caveats documented in CLAUDE.md.
  3. **[NO-GO path]** If live capture is not viable, the decision + rationale are recorded, ESPN stays explicitly unsupported via the API, and the **manual-entry fallback (D-09)** is verified as the ESPN path — no fragile half-build is merged.

## Notes

- **Biggest integration risk: player-ID mapping** (SLPR-03 / per-adapter). Each platform uses its own player id; our projections key on name/team/position. Sleeper's `/v1/players/nfl` registry is the bridge (~5MB, cache it); Yahoo and ESPN each need their own mapping. DST and kickers are the usual mismatch sources — handle explicitly.
- **Second risk: snake-order + slot math** (ENG-02). Drafts can be snake, linear, or auction, and may include traded picks. Pin slot/turn logic with fixtures from a real completed draft.
- **Adapter discipline (D-08) is the keystone.** If the engine ever reaches past the `DraftAdapter` interface into Sleeper specifics, Yahoo/ESPN expansion breaks. Phase 86 ENG-05 enforces this; treat any Sleeper import in the engine core as a defect.
- **ESPN is genuinely uncertain** — no official live API. Phase 89 leads with a spike, not a build; the honest floor is the manual-entry fallback (D-09). Do not promise ESPN live parity until the spike returns GO.
- **Yahoo cost is OAuth, not feasibility** — official API, `draft_results` works mid-draft. Budget time for token-refresh plumbing (reuse `yfpy`) and conservative polling (HTTP 999 throttling).
- Out of scope entirely: any website/chatbot surfacing of draft advice (separate from this milestone).
