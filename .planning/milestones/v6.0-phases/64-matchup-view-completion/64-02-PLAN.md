<objective>
Expose real NFL roster data via three FastAPI endpoints that the matchup view will consume in plan 64-04. This plan implements the MTCH-02 backend (defensive rosters with real NFL player names and stats) and the MTCH-04 backend (schedule-aware current week).

Purpose: The matchup view in `web/frontend/src/features/nfl/components/matchup-view.tsx` synthesizes defensive players from a slotHash and defaults the season/week to hardcoded 2026/1. Both fail the phase goal because they have no coupling to real NFL data. This plan builds the data path; plan 64-04 wires the frontend.

Output:
- `/api/teams/current-week` — returns `{season, week, source}` from today's date against the latest local schedule parquet.
- `/api/teams/{team}/roster?season&week&side` — returns real NFL players with `player_name`, `position`, `depth_chart_position`, `status`, `snap_pct_defense`, `snap_pct_offense`.
- Fallback logic: when 2026 data is absent, serve latest available season and set `fallback: true` in response metadata.
- Pytest suite covering happy path, fallback, and unknown-team 404.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-64-02-01 | Tampering | team path param (`/teams/{team}/roster`) | mitigate | Whitelist validation: compare against set loaded from roster df; raise ValueError for anything not in the set (already part of `load_team_roster`). FastAPI path params are strings — do not interpolate into shell or SQL. |
| T-64-02-02 | Info Disclosure | season/week Query | mitigate | Use `Query(..., ge=2016, le=2030)` and `Query(..., ge=1, le=22)` so FastAPI rejects out-of-range before service code runs. |
| T-64-02-03 | DoS | file system reads per request | mitigate | Parquet reads are small (<3k rows per roster snapshot, <2k per snap week). No pagination needed for MVP. Cache not added here; add if load spikes. |
| T-64-02-04 | Info Disclosure | fallback response leaking unexpected season | accept | Fallback flag is explicit in response body; documented in API-CONTRACT. Acceptable because there is no sensitive data — all rosters are public NFL data. |
| T-64-02-05 | Repudiation | no audit trail for data-fallback | accept | Existing services (games, projections) do not log per-request; structured logging is out of scope for v6.0. |
