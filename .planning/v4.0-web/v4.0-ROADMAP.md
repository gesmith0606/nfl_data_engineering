# v4.0 Web Platform — Roadmap

Phased delivery plan for the NFL web platform MVP. Phases use `W` prefix to distinguish from data engineering milestones (v3.x).

---

## Phase W1: FastAPI Backend

**Goal**: Serve projections and predictions as JSON from existing Parquet files.

**Requirements**: WEBAPI-01, WEBAPI-02, WEBAPI-03

**Work items**:
1. Create `api/` directory with FastAPI app structure:
   ```
   api/
     __init__.py
     main.py              # FastAPI app, CORS, lifespan
     config.py            # Settings from env vars
     routers/
       projections.py     # GET /api/v1/projections
       predictions.py     # GET /api/v1/predictions
       players.py         # GET /api/v1/players/search, /api/v1/players/{id}
     services/
       projection_service.py   # Read Parquet, filter, paginate
       prediction_service.py   # Read prediction Parquet
       player_service.py       # Player lookup and search
     models/
       schemas.py         # Pydantic response models
   ```
2. Projection endpoint reads from `data/gold/projections/season=YYYY/week=WW/*.parquet`
   - Uses `src/utils.py` → `download_latest_parquet()` pattern (adapted for local read)
   - Recalculates points for requested scoring format via `src/scoring_calculator.py`
3. Prediction endpoint reads from `data/gold/predictions/season=YYYY/week=WW/*.parquet`
4. Player search: in-memory index built at startup from latest projection Parquet
5. OpenAPI docs auto-generated at `/docs`
6. CORS configured for `http://localhost:3000` (dev) and Vercel domains (prod)
7. Health check at `GET /api/v1/health`
8. Local dev: `uvicorn api.main:app --reload --port 8000`

**Verification**:
- All endpoints return valid JSON matching Pydantic schemas
- `pytest tests/api/` covers happy path + edge cases (no data, invalid params)
- Projection endpoint returns different points for each scoring format
- Response times < 500ms on local Parquet reads

**Estimated effort**: 2-3 sessions

## Phase W2: Next.js Frontend — Projections

**Goal**: Weekly projections table with position filters and scoring toggle.

**Requirements**: WEB-01, WEB-04, WEB-06

**Work items**:
1. Initialize Next.js 14+ project in `web/` directory:
   ```
   web/
     app/
       layout.tsx          # Root layout, navbar, fonts
       page.tsx            # Redirect to /projections
       projections/
         page.tsx          # Server component: fetch + render table
         ProjectionsTable.tsx  # Client component: sort, filter, toggle
         ScoringToggle.tsx     # PPR / Half-PPR / Standard buttons
         PositionFilter.tsx    # All / QB / RB / WR / TE / K tabs
         WeekSelector.tsx      # Dropdown for week 1-18
       predictions/        # (placeholder, built in W3)
       players/
         [slug]/
           page.tsx        # (placeholder, built in W4)
     lib/
       api.ts              # Fetch wrapper for FastAPI
       types.ts            # TypeScript types matching API schemas
       constants.ts        # Team colors, position lists
     tailwind.config.ts
     next.config.ts
   ```
2. Projections page: server component fetches from FastAPI, renders table
3. Client-side interactivity:
   - Column sorting (click header to sort asc/desc)
   - Position filter tabs (All / QB / RB / WR / TE / K)
   - Scoring format toggle (re-fetches with `?scoring=` param)
   - Week selector dropdown
   - Search input (filters client-side or calls `/players/search`)
4. Pagination: "Load more" button or infinite scroll (50 players per page)
5. Responsive: sticky first column (player name) on horizontal scroll at mobile widths
6. Player name links to `/players/[slug]` (placeholder page until W4)
7. Bye week rows: grayed out with "BYE" text badge
8. Rookie rows: small "R" badge next to player name

**Verification**:
- Page renders with real projection data from local FastAPI
- Scoring toggle changes point values
- Position filters work correctly
- Table is usable on 375px-wide viewport (iPhone SE)
- Lighthouse performance score > 80

**Estimated effort**: 3-4 sessions

## Phase W3: Game Predictions Page

**Goal**: Display game predictions with matchup cards, edge tiers, and confidence.

**Requirements**: WEB-02

**Work items**:
1. Create `web/app/predictions/page.tsx` — server component fetching from predictions API
2. Matchup card component:
   - Team abbreviations with optional team color accents
   - Model spread/total vs Vegas spread/total
   - Edge value with color-coded tier badge (HIGH=green, MEDIUM=yellow, LOW=gray)
   - Pick summary line ("BAL covers", "OVER 48.7")
3. Confidence bar: horizontal bar showing edge magnitude relative to max edge that week
4. Sort: high-edge games appear first (largest absolute spread_edge + total_edge)
5. Week selector shared with projections (stored in URL params or layout state)
6. Responsive: single-column card layout on mobile, 2-column on md+

**Verification**:
- All games for a week render as cards
- Edge tier badges match threshold rules (>=3.0 high, >=1.5 medium)
- High-edge games sorted to top
- Cards readable on mobile viewport

**Estimated effort**: 1-2 sessions

## Phase W4: Player Detail + Accuracy

**Goal**: Individual player page and historical accuracy display.

**Requirements**: WEB-03, WEBAPI-04

**Work items**:
1. Backend: add `GET /api/v1/accuracy` endpoint in `api/routers/accuracy.py`
   - Reads pre-computed backtest results from `data/gold/backtest/` or accuracy table
   - Returns overall + per-position MAE, RMSE, correlation, bias
2. Backend: add `GET /api/v1/players/{player_id}` endpoint
   - Returns full projection detail including all projected stats
   - Includes opponent matchup ranking and vegas multiplier
3. Player detail page (`web/app/players/[slug]/page.tsx`):
   - Floor/ceiling range bar (horizontal, position of projected_points marked)
   - Projected stats table grouped by category (passing, rushing, receiving)
   - Opponent matchup info (opponent name, positional rank)
   - Vegas multiplier display
   - Recent performance (last 3 weeks if available)
4. Accuracy page (`web/app/accuracy/page.tsx`):
   - Overall metrics displayed as stat cards (MAE, RMSE, correlation)
   - Per-position breakdown table
   - Seasons and scoring format noted

**Verification**:
- Player detail shows correct projection breakdown
- Floor/ceiling bar renders proportionally
- Accuracy page displays real backtest numbers
- Slug-based routing works (e.g., `/players/josh-allen`)

**Estimated effort**: 2-3 sessions

## Phase W5: Database + Deployment

**Goal**: PostgreSQL for production reads, Vercel + AWS deployment.

**Requirements**: DATA-01, DATA-02, DATA-03, INFRA-01, INFRA-02, INFRA-03, INFRA-04

**Work items**:
1. PostgreSQL setup:
   - Create Supabase project (free tier)
   - Define tables: `projections`, `predictions`, `accuracy_snapshots` (schemas in REQUIREMENTS.md)
   - Alembic migration files in `api/migrations/`
2. Sync script: `scripts/sync_gold_to_db.py`
   - Reads latest Parquet from `data/gold/projections/` and `data/gold/predictions/`
   - Upserts into PostgreSQL using `psycopg2` or `asyncpg`
   - Stores all three scoring formats per player-week
   - Integrated into weekly pipeline (GHA workflow step)
3. FastAPI database integration:
   - SQLAlchemy async or raw asyncpg for queries
   - Fallback to Parquet reads if DB unavailable
   - Connection pooling via Supabase pooler endpoint
4. Vercel deployment:
   - Connect GitHub repo to Vercel
   - Set root directory to `web/`
   - Configure environment variables (`NEXT_PUBLIC_API_URL`)
   - Verify preview deploys on PRs
5. AWS Lambda deployment:
   - Package FastAPI with Mangum adapter
   - SAM template or CDK stack in `infra/`
   - API Gateway with custom domain (optional for MVP)
   - Environment variables via Parameter Store or Lambda config
6. CI/CD:
   - GitHub Actions workflow: lint + test frontend, deploy to Vercel
   - Separate workflow: test + deploy backend to Lambda
   - Weekly pipeline adds DB sync step after Parquet generation

**Verification**:
- Frontend loads from Vercel URL with real data from Lambda API
- API reads from PostgreSQL with < 100ms response time
- Weekly pipeline successfully syncs Parquet to DB
- Preview deploys work on PR branches
- Environment separation (dev/staging/prod) confirmed

**Estimated effort**: 3-4 sessions

## Phase W6: Polish + Dark Mode

**Goal**: Dark/light mode, team colors, SEO, API key auth.

**Requirements**: WEB-05, WEBAPI-05

**Work items**:
1. Dark/light mode:
   - Tailwind `darkMode: 'class'` configuration
   - Toggle button in navbar (sun/moon icon)
   - System preference detection via `prefers-color-scheme`
   - Preference persisted in `localStorage`
   - All pages styled for both modes
2. NFL team colors:
   - `web/lib/teamColors.ts` — 32-team map of `{primary: string, secondary: string}`
   - Team badge component with colored background
   - Matchup card borders use home team primary color
   - Player detail page header uses team colors
3. SEO:
   - Dynamic `<title>` and `<meta description>` per page
   - Projections page: "Week 1 2026 Fantasy Projections - PPR Rankings"
   - Player page: "Josh Allen Week 1 Projection - 22.4 pts (Half-PPR)"
   - JSON-LD structured data for projection pages (SportsEvent schema)
   - Open Graph meta tags for social sharing
   - `sitemap.xml` generated at build time
   - `robots.txt` allowing all crawlers
4. API key authentication:
   - FastAPI middleware checking `X-API-Key` header
   - Keys from `API_KEYS` environment variable
   - 401 JSON response on missing/invalid key
   - Rate limiting: 100 req/min per key (in-memory counter or API Gateway throttle)
   - Frontend passes key via `NEXT_PUBLIC_API_KEY` (acceptable for MVP; server-side proxy later)

**Verification**:
- Dark mode toggles correctly, persists across page loads
- Team colors render on player rows and matchup cards
- Lighthouse SEO score > 90
- Google can index server-rendered projection pages
- API rejects requests without valid key
- Rate limit returns 429 after threshold

**Estimated effort**: 2-3 sessions
