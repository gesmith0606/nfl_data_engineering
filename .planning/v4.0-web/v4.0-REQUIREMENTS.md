## API Layer

### WEBAPI-01: Player Projections Endpoint

FastAPI endpoint serving weekly player projections as JSON.

```
GET /api/v1/projections
  ?season=2026          (required, int)
  &week=1               (required for weekly; omit for preseason)
  &scoring=half_ppr     (optional, default: half_ppr; enum: ppr, half_ppr, standard)
  &position=QB          (optional, filter; enum: QB, RB, WR, TE, K)
  &team=KC              (optional, filter; 2-3 letter team abbrev)
  &limit=50             (optional, default: 200, max: 500)
  &offset=0             (optional, for pagination)
```

Response shape:
```json
{
  "season": 2026,
  "week": 1,
  "scoring_format": "half_ppr",
  "total_players": 312,
  "projections": [
    {
      "player_id": "00-0036442",
      "player_name": "Josh Allen",
      "position": "QB",
      "team": "BUF",
      "projected_points": 22.4,
      "projected_floor": 12.3,
      "projected_ceiling": 32.5,
      "is_bye_week": false,
      "is_rookie_projection": false,
      "projected_stats": {
        "passing_yards": 274.2,
        "passing_tds": 1.8,
        "interceptions": 0.7,
        "rushing_yards": 38.1,
        "rushing_tds": 0.4
      },
      "rank": 1,
      "position_rank": 1
    }
  ],
  "metadata": {
    "generated_at": "2026-04-01T08:00:00Z",
    "model_version": "v3.0",
    "data_through_week": 0
  }
}
```

Source: `src/projection_engine.py` → `generate_weekly_projections()` and `generate_preseason_projections()`.

Acceptance criteria:
- Returns projections for all skill positions (QB, RB, WR, TE, K)
- Scoring format changes recalculate points using `src/scoring_calculator.py`
- Position and team filters work independently and combined
- Pagination via limit/offset
- Response time < 500ms (Parquet read) or < 100ms (PostgreSQL)

### WEBAPI-02: Game Predictions Endpoint

```
GET /api/v1/predictions
  ?season=2026          (required, int)
  &week=1               (required, int)
```

Response shape:
```json
{
  "season": 2026,
  "week": 1,
  "games": [
    {
      "game_id": "2026_01_KC_BAL",
      "home_team": "BAL",
      "away_team": "KC",
      "model_spread": -3.2,
      "model_total": 48.7,
      "vegas_spread": -2.5,
      "vegas_total": 47.5,
      "spread_edge": -0.7,
      "total_edge": 1.2,
      "spread_confidence_tier": "low",
      "total_confidence_tier": "low",
      "spread_pick": "BAL -3.2",
      "total_pick": "OVER 48.7",
      "model_version": "P30 Ensemble"
    }
  ],
  "metadata": {
    "prediction_timestamp": "2026-04-01T08:00:00Z",
    "model_version": "P30 Ensemble",
    "edge_thresholds": {
      "high": 3.0,
      "medium": 1.5
    }
  }
}
```

Source: `scripts/generate_predictions.py` output columns (see `OUTPUT_COLUMNS`), `src/ensemble_training.py`.

Acceptance criteria:
- Returns all games for the requested week
- Edge tiers match existing thresholds (>=3.0 high, >=1.5 medium, else low)
- Spread sign convention preserved (negative = home favored)
- Response time < 200ms

### WEBAPI-03: Player Search and Filter

```
GET /api/v1/players/search
  ?q=josh               (required, min 2 chars; searches player_display_name)
  &position=QB          (optional filter)
  &team=BUF             (optional filter)
  &limit=20             (optional, default: 20)
```

Response:
```json
{
  "results": [
    {
      "player_id": "00-0036442",
      "player_name": "Josh Allen",
      "position": "QB",
      "team": "BUF"
    }
  ],
  "total": 1
}
```

Acceptance criteria:
- Case-insensitive substring match on player name
- Returns within 100ms
- Minimum 2-character query to prevent full-table scans

### WEBAPI-04: Historical Accuracy Endpoint

```
GET /api/v1/accuracy
  ?seasons=2022,2023,2024   (optional, default: last 3 seasons)
  &scoring=half_ppr          (optional, default: half_ppr)
```

Response:
```json
{
  "overall": {
    "mae": 4.91,
    "rmse": 6.72,
    "correlation": 0.510,
    "bias": -0.60,
    "player_weeks": 11183
  },
  "by_position": {
    "QB": {"mae": 6.58, "rmse": 8.92, "bias": -2.20, "player_weeks": 1520},
    "RB": {"mae": 5.06, "rmse": 6.81, "bias": -0.37, "player_weeks": 3200},
    "WR": {"mae": 4.85, "rmse": 6.54, "bias": -0.28, "player_weeks": 4100},
    "TE": {"mae": 3.77, "rmse": 5.12, "bias": -0.63, "player_weeks": 2363}

    "weeks_per_season": "3-18",
```

Source: `scripts/backtest_projections.py` output.

Acceptance criteria:
- Serves pre-computed backtest results (not computed on-the-fly)
- Updated when backtest is re-run
- Includes per-position breakdown

---

### WEBAPI-05: API Key Authentication

- All endpoints require `X-API-Key` header
- Keys stored in environment variable (`API_KEYS` comma-separated list)
- MVP: single shared key for frontend-to-backend auth
- 401 response on missing/invalid key
- Rate limit: 100 requests/minute per key (via middleware)
- No user-level auth in v4.0; API key is service-to-service

### WEB-04: Responsive Design

- Mobile breakpoint: 640px (Tailwind `sm`)
- Tablet breakpoint: 768px (Tailwind `md`)
- Desktop: 1024px+ (Tailwind `lg`)
- Projections table on mobile: horizontal scroll with sticky player name column

- Default: follow system `prefers-color-scheme`

- Dark palette: slate-900 background, slate-100 text
- Light palette: white background, slate-900 text
- NFL team color accents:
  - Team badge backgrounds on player rows and detail pages
  - Matchup card borders colored by home team
  - Static map of 32 team → {primary: hex, secondary: hex} in a `teamColors.ts` file

### WEB-06: Scoring Format Toggle

- Three-button toggle group: PPR | Half-PPR | Standard
- Default: Half-PPR (matches project default)

- Active format highlighted; persisted in `localStorage` across sessions
- Points column, rank, and position rank all update on toggle
- Player detail page respects the selected format

## Data Pipeline

### DATA-01: Write Projections to PostgreSQL

- `scripts/sync_gold_to_db.py` reads latest Parquet from `data/gold/projections/` and upserts into `projections` table
- Schema:
  ```sql
  CREATE TABLE projections (
    id            SERIAL PRIMARY KEY,
    player_id     VARCHAR(20) NOT NULL,
    player_name   VARCHAR(100) NOT NULL,
    position      VARCHAR(5) NOT NULL,
    team          VARCHAR(5) NOT NULL,
    season        INT NOT NULL,
    week          INT NOT NULL,
    scoring_format VARCHAR(20) NOT NULL,
    projected_points DECIMAL(6,2),
    projected_floor  DECIMAL(6,2),
    projected_ceiling DECIMAL(6,2),
    is_bye_week   BOOLEAN DEFAULT FALSE,
    is_rookie     BOOLEAN DEFAULT FALSE,
    projected_stats JSONB,
    created_at    TIMESTAMP DEFAULT NOW(),
    UNIQUE(player_id, season, week, scoring_format)
  );
  ```
- Runs after `generate_projections.py` in weekly pipeline
- Stores all three scoring formats per player-week

---

### DATA-02: Write Predictions to PostgreSQL

- Same sync script writes predictions to `predictions` table
- Schema:
  ```sql
  CREATE TABLE predictions (
    id              SERIAL PRIMARY KEY,
    game_id         VARCHAR(30) NOT NULL,
    season          INT NOT NULL,
    week            INT NOT NULL,
    home_team       VARCHAR(5) NOT NULL,
    away_team       VARCHAR(5) NOT NULL,
    model_spread    DECIMAL(5,2),
    model_total     DECIMAL(5,2),
    vegas_spread    DECIMAL(5,2),
    vegas_total     DECIMAL(5,2),
    spread_edge     DECIMAL(5,2),
    total_edge      DECIMAL(5,2),
    spread_confidence VARCHAR(10),
    total_confidence  VARCHAR(10),
    model_version   VARCHAR(30),
    created_at      TIMESTAMP DEFAULT NOW(),
    UNIQUE(game_id, model_version)
  );
  ```

---

### DATA-03: Historical Data Retention

- Projections and predictions retained indefinitely (no TTL)
- Backtest summary stats stored in `accuracy_snapshots` table:
  ```sql
  CREATE TABLE accuracy_snapshots (
    id              SERIAL PRIMARY KEY,
    seasons         VARCHAR(50) NOT NULL,
    scoring_format  VARCHAR(20) NOT NULL,
    position        VARCHAR(5),
    mae             DECIMAL(6,3),
    rmse            DECIMAL(6,3),
    correlation     DECIMAL(6,4),
    bias            DECIMAL(6,3),
    player_weeks    INT,
    computed_at     TIMESTAMP DEFAULT NOW()
  );
  ```
- Updated when `scripts/backtest_projections.py` runs

## Infrastructure

### INFRA-01: Vercel Deployment

- Next.js deployed to Vercel from `web/` subdirectory (monorepo root deploy)
- GitHub integration: auto-deploy on push to `main`; preview deploy on PRs
- Environment variables: `NEXT_PUBLIC_API_URL`, `API_KEY`
- Build command: `cd web && npm run build`
- ISR revalidation: 300 seconds (5 minutes) for projection pages

---

### INFRA-02: FastAPI on AWS

- Option A (MVP): AWS Lambda + API Gateway via Mangum adapter
  - Cold start ~2s acceptable for MVP traffic
  - Deploy via SAM or CDK
- Option B (scale-up): ECS Fargate if Lambda latency becomes an issue
- API Gateway custom domain: `api.nflprojections.com` (or similar)
- Environment variables: `DATABASE_URL`, `API_KEYS`, `PARQUET_DIR`

---

### INFRA-03: PostgreSQL Database

- Supabase free tier (500MB, 2 projects) for MVP
  - Connection string via `DATABASE_URL` env var
  - Direct connection for FastAPI; pooler connection for serverless
- Upgrade path: AWS RDS PostgreSQL when traffic or data exceeds free tier
- Migration tool: Alembic (Python, consistent with existing stack)

---

### INFRA-04: Environment-Based Configuration

- Three environments: `dev` (local), `staging` (Vercel preview + Lambda alias), `prod`
- Config via environment variables (no config files with secrets)
- Required env vars:

| Variable | Dev | Staging | Prod |
|----------|-----|---------|------|
| `DATABASE_URL` | `postgresql://localhost:5432/nfl_dev` | Supabase staging | Supabase prod |
| `API_KEYS` | `dev-key-123` | `staging-key-xxx` | `prod-key-yyy` |
| `PARQUET_DIR` | `./data/gold` | S3 path | S3 path |
| `NEXT_PUBLIC_API_URL` | `http://localhost:8000` | Lambda staging URL | `https://api.domain.com` |
| `CORS_ORIGINS` | `http://localhost:3000` | Vercel preview URLs | `https://domain.com` |
